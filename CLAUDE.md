# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Python environment

> **IMPORTANT: Always install packages into `.venv` — never system-wide.**

A virtual environment lives at `.venv/`. Activate it before running any Python command:

```bash
source .venv/bin/activate
```

To set up from scratch:

```bash
python -m venv .venv
source .venv/bin/activate
pip install -e ".[dev]"
```

## Architecture

The pipeline runs in six sequential stages, each in its own module:

```
Telegram (Telethon) → scraper.py → db.py ← analyzer.py (LM Studio) → triage.py → generator.py
                                                                                        ↓
                                                              briefings/YYYY-MM-DD/TheDailyTelegram_YYYY-MM-DD_HHMMSS.pdf
                                                                                        ↓
                                                                          synthesiser.py (LM Studio)
                                                                                        ↓
                                                                   intelligence front page prepended to PDF
```

**`config.py`** — Single `AppConfig` Pydantic model loaded from `config.yaml`. Use `load_config(path, env_override=True)` everywhere; it reads `TG_API_ID` / `TG_API_HASH` / `LM_API_TOKEN` from env (`.env` is auto-loaded by `main.py` via `python-dotenv`). Every config model (including all nested sub-models) has `extra="forbid"` so YAML typos fail loudly at any nesting level. All fields in the config model are actively used — see `config.yaml.example` for the full reference with comments. `ChannelConfig.credibility` (default `1.0`) is a per-channel credibility prior multiplied into the composite score alongside `priority`. `TriageConfig.rumor_penalty` (default `0.7`) down-weights posts with `category == "Rumor"`. `TriageConfig.dedup_jaccard_threshold` / `dedup_entity_overlap_count` / `dedup_entity_cluster_overlap_count` (defaults `0.28`/`3`/`4`) configure `triage._find_duplicate()`'s dedup thresholds.

**`db.py`** — Owns the SQLite schema and all SQL. The two domain dataclasses (`PostRecord`, `AnalysisRecord`) live here. `AnalysisRecord` includes `title` (LLM-generated 5-10 word headline) and `threat_level` (one of `CRITICAL`, `HIGH`, `MODERATE`, `LOW`). `insert_post` returns `None` on duplicate (UNIQUE on `channel_id + message_id`) — callers use this to skip already-seen posts. `get_unanalysed_posts` drives the analyzer's work queue. `reset_all_cursors()` sets every channel cursor to 0 (used by `--since`). Safe migrations run at `init_schema()` time using `PRAGMA table_info` before any `ALTER TABLE` — safe on both fresh and existing databases. `get_days_posts_with_analyses()` and `get_posts_with_analyses_in_range(start_date_str, end_date_str)` (inclusive `DATE(p.timestamp) BETWEEN`) both build `(PostRecord, AnalysisRecord)` pairs via the shared `_rows_to_pairs()` helper. The `intel_assessments` table (`date` PK, `intel_json`) persists each day's synthesised intelligence JSON via `save_intel_assessment(date_str, intel)` (upsert) / `get_intel_assessment(date_str)` (returns `None` if absent), feeding next-day continuity in `synthesiser.py`. `init_schema()` also creates `idx_analyses_post_id` and `idx_posts_timestamp` indexes; `__init__` enables `PRAGMA foreign_keys=ON` and (for non-`:memory:` paths) chmods the DB file to `0600` via `secure_file()`. `Database` supports `close()` and use as a context manager (`__enter__`/`__exit__`).

**`scraper.py`** — Telethon async context manager. `Scraper.scrape_channel()` fetches messages since the last cursor (`offset_id=last_seen_id`); when the cursor is 0 (first run or after reset), uses `offset_date = now - lookback_seconds` instead. `iter_messages` is called with `limit=None` — no artificial cap, so a long backlog (e.g. after `--since`) is fully drained in one run. Skips `MessageService` objects (system events), downloads photos (with one retry), inserts to DB, and advances the per-channel cursor. `get_entity()` and the whole iteration are inside one try/except — a renamed/deleted channel or other error logs and returns whatever was collected so far instead of propagating; `run_batch` also wraps each `scrape_channel` call so one bad channel never aborts the rest of the batch.

**`analyzer.py`** — Uses the `openai` package against LM Studio's OpenAI-compatible REST endpoint (`/v1/chat/completions`), with a `timeout=120` client to bound a hung server. `SYSTEM_PROMPT` defines per-axis 1–5 scoring rubrics, category definitions, summary style (describe the event, never "the post"/"the image"; attribute unverified claims; use titles/offices exactly as the post gives them — never "corrected" from training knowledge, which once turned "President Trump" into "Former President Trump"), canonical entity rules (no news-agency/platform source credits), English translation of non-English text visible in images, and the threat-level rubric. `PostAnalysis` is a Pydantic schema with `title`, `summary`, four 1-5 score fields, `category`, `key_entities`, `image_description`, and `threat_level`. `threat_level` is validated by a `@field_validator` — anything outside `{CRITICAL, HIGH, MODERATE, LOW}` defaults to `MODERATE`. Structured output via `client.beta.chat.completions.parse(response_format=PostAnalysis)` with a plain text fallback (no `response_format` parameter — LM Studio only supports `json_schema` or `text`). Fallback response strips markdown fences before JSON parsing, then falls through to regex extraction. `_sanitize()` cleans title, entities, and image description (`_clean_image_insights` also rejects "none provided"/"no video provided" variants), blanks `title`/`summary` if they match `_REFUSAL_RE` (LLM meta-talk like "the user provided", "no content provided", "as an AI"), then HTML-escapes `title`, `summary`, `key_entities`, and `image_description` via `escape_html()` (utils.py) so LLM output can never inject markup into the rendered briefing. `analysis_to_record(post_id, analysis, model_used)` builds an `AnalysisRecord` from a `PostAnalysis` and is the single place both `process_unanalysed` (batch) and `run_daemon` (main.py) construct records, so fields like `title` can't diverge between the two paths again. `process_unanalysed()` first runs `_server_reachable()` — a 10s `models.list()` preflight probe — and aborts with everything left queued if LM Studio is down (a dead server fails in seconds, not hours). It then runs analyses concurrently up to `lmstudio.max_concurrent_analyses` and returns `(analysed_count, skipped_count)`; posts with `len(text.strip()) < MIN_CONTENT_CHARS` (30) and no media are skipped before calling the LLM, counted in `skipped_count`, and recorded with `category="Skipped"` (filtered out in `triage.py`) so they're never retried. `analyze_post()` propagates its final error instead of fabricating an empty analysis — a failed post is logged, left unanalysed, and retried on the next run (the regex fallback is only used for received-but-unparseable responses). The analyzer's OpenAI client uses `max_retries=0` since `analyze_post` has its own 3-attempt backoff loop.

**`triage.py`** — Pure function `triage(pairs, config, channel_priorities, channel_credibilities) -> BriefingContent`. Posts with `analysis.category == "Skipped"` (see `analyzer.py` content gate) are excluded before scoring. Badge guard: a `CRITICAL` threat level with `category == "Rumor"` or `credibility_score <= 2` is demoted to `HIGH` at triage time (CRITICAL means "confirmed … underway"; this also stops unsubstantiated posts hijacking guaranteed lead slots, and applies to already-stored analyses). Composite score: `(0.4*importance + 0.3*urgency + 0.2*credibility + 0.1*relevance) × channel_priority × channel_credibility`, keyword-boosted (first match only, capped at 5.0), down-weighted by `rumor_penalty` when `category == "Rumor"`, multiplied by `threat_multipliers[threat_level]` (defaults CRITICAL 1.15 / HIGH 1.05 / MODERATE 1.0 / LOW 0.85, so badges nudge rank), then multiplied by a recency factor from `_recency_multiplier()` — an exponential half-life decay (`recency_half_life_hours`, default 24h) floored at `recency_floor` (default 0.7) so very old posts are demoted but never zeroed out. Recency is anchored to `min(now, end of the briefing day)` so retrospective runs (`--analyse --since <past date>`) reproduce the original day's ranking instead of decaying everything to the floor. After scoring, posts are clustered rather than dropped: `_find_duplicate()` matches on Jaccard word-overlap ≥ `dedup_jaccard_threshold` (default 0.28) on summary/title within `dedup_summary_window_secs` (default 6h), OR ≥`dedup_entity_overlap_count` (default 3) shared named entities within `dedup_window_secs`, OR ≥`dedup_entity_cluster_overlap_count` (default 4) shared entities within `entity_cluster_window_secs` — entity comparisons are normalised via `_normalize_entity()` / `_ENTITY_ALIASES` (e.g. "U.S." / "US" / "USA" → "united states", "IDF" → "israel defense forces") so naming variants match. A matched duplicate is appended as a `CorroborationRef` (channel_slug, message_id, timestamp) on the higher-scored representative's `TriagedPost.corroborations` instead of being discarded. Representatives with corroborations get a multiplicative score boost — `score *= min(1 + corroboration_weight * n, corroboration_cap)` where `n` is the number of *distinct other channels* among the corroborations (same-channel reposts are merged but earn no boost; repeat posts from one other channel count once) — and the kept list is re-sorted afterwards since boosting can reorder items. `posts_clustered` (on `BriefingContent`) is the count of duplicates merged this way. Survivors split into `main_items` / `appendix_items` at `min_composite_score`; if fewer than `min_main_items` (default 15) clear the threshold, the highest-scoring appendix items are promoted to fill the shortfall, then `main_items` is capped at `max_main_items` (overflow goes to appendix). `BriefingContent` also carries `executive_items` (the Lead Reports) — all `CRITICAL`-threat-level items first (from main *and* appendix, so a low-scoring CRITICAL is still promoted), topped up with the highest-scored non-critical `main_items` to 10 total; criticals are never truncated, so the list grows past 10 when criticals alone exceed it. `executive_items` overlaps `main_items`/`appendix_items` by design — the template de-duplicates at render time, while the synthesiser keeps consuming the full `main_items`. `channel_links` (slug → bare username) drives deep links and `category_counts` (category → post count) feeds the statistics table. `posts_scraped`/`posts_analysed`/`posts_skipped` are zero-valued by `triage()` and set externally by `main.generate_daily_briefing()`.

**`generator.py`** — `render_markdown(content)` resolves each executive (lead) item's `post.media_paths` to absolute paths (`Path(p).resolve()`) before rendering, then renders via Jinja2 (`templates/briefing.md.j2`); `generate_briefing(..., pdf=True)` then runs `markdown-pdf` (PyMuPDF backend) with `Section(md_text, root="/")` — the section root must be `/` because `fitz.Story` resolves image srcs against it and the default `"."` silently drops absolute paths (PDF shows an `[image]` placeholder instead). PDFs are saved to `{output_dir}/{YYYY-MM-DD}/TheDailyTelegram_{YYYY-MM-DD}_{HHMMSS}.pdf`; the `.md` source goes alongside it. Threat level badges are HTML `<span>` elements with inline `color:` styles (MuPDF renders text colour but not background-colour on inline elements). Images are emitted as `<img src="…" width="300">` tags, NOT markdown `![]()` + CSS `max-width` — `fitz.Story` ignores `max-width`, and a natural-size image landing at a page break makes Story silently drop that element *and the blocks after it* (this once swallowed an entire section). The template renders three *disjoint* tiers (computed via Jinja `reject('in', lead)` so no story prints twice): **Lead Reports** — `content.executive_items` in full detail (header, byline with channel/score/link, summary, entities, corroborations, video/image block, up to 3 embedded images); **Other Developments** — remaining `main_items` as compact `.dev-list` entries (badge, category, bold headline, channel/time/clamped-score/link meta, then the summary on its own line and a "Corroborated by N other channels" line with links when present); **In Brief** — remaining `appendix_items` in the two-column compact layout, with cross-channel corroboration abbreviated to a `+N corrob` marker in the score parens. There are no per-channel sections. Each lead item's corroborations split by channel: cross-channel refs render as "Corroborated by N other channel(s)" (N = distinct channels, deep links via `channel_links`), same-channel refs as a separate "Related posts from this channel" line with time-stamped links. The image/video block shows a "Video" line (with `▶ Watch on Telegram` link) when `item.post.has_video`, an "Image" line when `item.analysis.image_insights` is set, or nothing for text-only posts. Displayed per-item scores are clamped to 5.0 (the corroboration boost can push the internal composite above 5). Statistics render as a compact `.stats-block` div (Items with lead/other/in-brief split / Pipeline / Categories / Channels lines; the Pipeline line appears only when `posts_scraped`/`posts_analysed`/`posts_skipped` is non-zero, i.e. after a `--batch` run). The template ends with a static `.smallprint` "Reader's Key" explaining sections, scoring, dedup and threat levels — template boilerplate never touched by the LLM; its only dynamic value is the configured score threshold, interpolated from `content.min_composite_score` (set by `triage()` from config). CSS at `templates/briefing.css`.

**`synthesiser.py`** — Intelligence front page generator. `_triaged_to_dicts(main_items)` converts a list of `TriagedPost` objects into the dict format consumed by `synthesise()`, including `message_id` (used for source deep links). `synthesise(config, posts, trends=None, previous_intel=None)` calls LM Studio (temp=0.2, max_tokens=12000 — reasoning models intermittently burn thousands of tokens on deliberation before emitting JSON, and a too-small budget surfaces as `finish_reason='length'` with truncated or empty content; the model has 32k context so the budget is deliberately generous — `timeout=300`, no `response_format`) and strips markdown fences before parsing. `_SYNTHESIS_SYSTEM`/`_SYNTHESIS_INSTRUCTIONS` instruct grounding (no invented events/numbers/actors, single-source claims marked unconfirmed), title/office fidelity (as the reports give them, not training knowledge), citation fidelity, themes as multi-report patterns, and canonical actor names (no news agencies/platforms); keep these prompts terse — verbose normative instructions made gemma-4-12b over-deliberate past its token budget. Each post is sent with a 1-based `index`; the prompt asks the model to cite supporting report indices as `"sources"` on each `key_themes`/`signals_and_warnings` item, and to set `"continuity"` (`new`/`confirmed`/`escalating`/`retired`) on each theme relative to `previous_intel["key_themes"]` if provided. `_format_trends(trends)` renders a "N-DAY MENTION TRENDS" block (entity + category prior→today counts, from `trends.py`) and `_format_previous_themes(previous_intel)` renders yesterday's themes for continuity — both return `""` when there's no data, and are appended to the user message before `_SYNTHESIS_INSTRUCTIONS`. `_validate_intel(data)` checks all four required keys, their sub-structure, and that each `key_themes`/`signals_and_warnings` item has a `"sources"` list. `_sanitize_intel()` HTML-escapes every text field via `escape_html()` (utils.py), filters `sources` to ints, and validates `continuity` against the four allowed values (defaulting to `"new"`). `_resolve_sources(items, posts, channel_links)` maps each item's `sources` indices (1-based, bounds-checked) back to `posts` entries, producing `source_links` (`channel_slug`, `time_str`, optional t.me `link`) for rendering. `_render_front_page_md(intel, date, posts=None, channel_links=None, emerging_entities=None)` renders `templates/intel_front_page.md.j2` — themes/signals show their `source_links` as footnotes and non-`"new"` `continuity` as an annotation; `emerging_entities` (entities mentioned today but absent from the prior 7 days, from `trends.py`) render as an "Emerging Actors / Topics" section (a single condensed `·`-separated line, `.emerging-list`) when non-empty. `_prepend_pdf()` uses `pypdf` to merge front page + briefing, writing to a temp file then atomically replacing the original with `os.replace()`; it is idempotent — it stores the front page's page count in the briefing PDF's `/IntelPages` metadata key, and on a subsequent call strips that many leading pages from the briefing before re-prepending, so re-running `--analyse` replaces rather than stacks the front page. `run_analysis(config, target_date, main_items=None)` is the async entry point — always opens the `Database` (closed via `try/finally`); when `main_items` is provided (passed from `run_batch`), synthesises directly from that triaged post list, when `main_items=None` (standalone `--analyse`) re-runs `get_days_posts_with_analyses()` + `triage()` to reconstruct the same post set as the main briefing. It then loads the last `trends.TREND_WINDOW_DAYS` (7) days via `get_posts_with_analyses_in_range()` to compute `trends` (`compute_trends()`), loads yesterday's `previous_intel` via `get_intel_assessment()`, calls `synthesise()`, and on success persists the result via `save_intel_assessment()` before rendering. Finds the latest `TheDailyTelegram_*.pdf` in the date dir, fails gracefully if LM Studio is unreachable, and never corrupts the existing PDF.

**`trends.py`** — Pure function `compute_trends(history, target_date) -> dict` over a multi-day list of `(PostRecord, AnalysisRecord)` pairs (typically the last `TREND_WINDOW_DAYS=7` days from `db.get_posts_with_analyses_in_range()`). Splits posts into "today" (`post.timestamp.date() == target_date`) vs "prior days" buckets, counting normalised entities (filtered through `utils.clean_entities`, then `triage._normalize_entity`) and categories in each; `category == "Skipped"` sentinel rows are excluded. Returns `entity_deltas`/`category_deltas` (`{"entity"|"category", "today_count", "prior_count"}`, sorted by today−prior descending, entities capped at `TOP_N_ENTITIES=10`) and `emerging_entities` — normalised entities with a non-zero today count and zero prior-day count, excluding `_EMERGING_STOPLIST` (newswire/photo credits and social platforms — sources, not actors) and entities starting with a digit (`_LEADING_DIGIT`: unit designators, years), sorted by today's frequency then alphabetically, capped at `MAX_EMERGING_ENTITIES=15`, and empty when there is no prior-day baseline at all (otherwise day one would flag everything). The template title-cases them for display.

**`utils.py`** — `escape_html(text)` escapes `&`, `<`, `>` (via `html.escape(text, quote=False)`) so LLM-derived text can't inject markup into the rendered Markdown/HTML output. `clean_entities()` and `_ENTITY_GARBAGE` filter junk/JSON-artifact entity strings (unchanged). `secure_file(path)` chmods a file to `0600`, used on the SQLite DB and Telethon `.session` files to keep credentials/data owner-only readable.

**`main.py`** — CLI entry point. `generate_daily_briefing(config, target_date, db, posts_scraped=0, posts_analysed=0, posts_skipped=0)` returns `(pdf_path, BriefingContent)`, setting those three counters on `content` after triage (the fourth, `posts_clustered`, is set inside `triage()` itself). `run_batch()` wires full scrape→analyse→triage→generate→synthesise pipeline, accumulating `total_scraped` across channels and unpacking `process_unanalysed()`'s `(analysed_count, skipped_count)`, then passes all three into `generate_daily_briefing()`; calls `run_analysis()` at the end, passing `content.main_items` so the Intel Assessment synthesises from the same triaged post set as the main briefing. All "today" values use `datetime.now(timezone.utc).date()` — the briefing day is always the UTC calendar date, regardless of host timezone. `--analyse` flag runs synthesis standalone (useful after `--generate`); accepts `--since` to target a specific date. `--since TIME` flag (with `--batch`) resets all cursors and sets `lookback_seconds` in memory (accepts `HH:MM`, `YYYY-MM-DD`, `YYYY-MM-DDTHH:MM`). `run_daemon()` listens for live Telethon events, downloads media, analyses inline (including `threat_level`), and runs `schedule_daily_generation()` as a background task with a `done_callback` that logs (`"Daily generation scheduler crashed"`, with traceback) if the task raises, so a scheduler crash is visible instead of silently vanishing. `run_daemon()` calls `secure_file()` (utils.py) on the `.session` file after `client.start()` to chmod it to `0600`. `schedule_daily_generation()` uses `generation.timezone` (IANA name, e.g. `"Europe/London"`) to interpret `generate_at` (the wall-clock trigger time), but the briefing date itself is computed via UTC `date()`; opens its own `Database` per iteration (closed via `try/finally`) and also calls `run_analysis()` after each nightly generation so the Intel Assessment is produced in daemon mode too. `purge_old_media()` removes date dirs older than `retention_days`. Config is only loaded after an action flag is confirmed (so `--help` works without a `config.yaml`).

## Data flow details

- **Idempotency**: `UNIQUE(channel_id, message_id)` in SQLite. Re-running the same window never duplicates records.
- **Cursor tracking**: `channel_cursors` table stores `last_seen_id` per channel. Scraper uses `offset_id=last_seen` on `iter_messages`. When cursor is 0, falls back to `offset_date` based on `lookback_seconds`. `--since` resets all cursors to 0 for a one-off historical lookback — do NOT use it for routine re-runs, as it forces Telegram to re-serve already-seen messages (wasted API quota). Plain `--batch` uses the cursor and only fetches new posts.
- **Daemon is live-only**: `run_daemon()` processes messages that arrive while running — it never backfills history. Always run `--batch` before switching to `--daemon` to catch up on historical posts.
- **Media**: Downloaded to `./data/media/<channel_slug>/<YYYY-MM-DD>/<message_id>.jpg`. Path constructed by `media_path_for()`.
- **`key_entities`**: Stored as JSON text in SQLite; deserialized back to `list[str]` in `get_days_posts_with_analyses`.
- **Output paths**: `briefings/YYYY-MM-DD/TheDailyTelegram_YYYY-MM-DD_HHMMSS.pdf` (unique per run) and `briefings/YYYY-MM-DD/briefing_YYYY-MM-DD.md` (overwritten on each run, stable name).
- **LM Studio compatibility**: Do not use `response_format={"type": "json_object"}` — this build only accepts `json_schema` or `text`. The structured output path uses `client.beta.chat.completions.parse()` with a Pydantic model; the fallback and synthesiser use plain `chat.completions.create()` with no `response_format` and parse the text response directly.

## Testing notes

- DB tests use `":memory:"` SQLite — the `db` fixture in `conftest.py` initialises schema fresh per test.
- `scraper.py` has no live-Telethon integration tests (requires credentials); `media_path_for()` is unit-tested and `scrape_channel`'s error-isolation / uncapped-iteration behaviour is covered with a monkeypatched client.
- `analyzer.py` tests construct `PostAnalysis` and message payloads locally (no LM Studio server needed). Full end-to-end runs against live Telegram/LM Studio are not covered by the suite.
- `asyncio_mode = "auto"` in `pyproject.toml` — async test functions work without decorators.
- 173 tests across 10 files; all pass with `pytest`.

## Runtime requirements

- **LM Studio** must be running (local or remote) with a VLM loaded. Set `lmstudio.server_host` and `lmstudio.server_port` in `config.yaml`. If authentication is enabled, set `LM_API_TOKEN` in `.env`.
- **`.env`** is auto-loaded at startup via `python-dotenv`. Supported vars: `TG_API_ID`, `TG_API_HASH`, `LM_API_TOKEN`. These override the corresponding YAML fields.
- **Telegram session**: First run will prompt for a phone number/code; the session is persisted to `<session_name>.session` (gitignored).
- `config.yaml` is gitignored. Copy `config.yaml.example` and fill in `api_id` and `api_hash`.
